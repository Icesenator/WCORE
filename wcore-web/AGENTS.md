## Architecture Cache (3 couches)

```
Couche 1 — L1 CacheService (volatile, 2h TTL)
  Clés: DEX:{chainSlug}:{contract}, GT:{network}:{contract}, LLAMA:{id}
  Prix en USD. Lecture rapide. Contourné par skipL1 sur forceFull (v4.13.9+)

Couche 2 — GlobalPriceCache (ScriptProperties, 6h staleness)
  Clés priceMap: {contract} → prix EUR
  Clés entries: {chainId}:{contract} → { price, ts, src }
  Partagé entre toutes les chaînes. Sauvé en fin de cycle.

Couche 3 — WalletCache (ScriptProperties packed, 14j TTL)
  Format compact v5: [contract, balance, symbol, name, decimals]
  Virtualisé dans GLOBAL_WALLET_CACHE (hash-based, 495KB max)
  Contient: assets, priceMap (EUR), priceTsMap, balanceTsMap, scanStats
```

**forceFull (C1=TRUE)** : clear WalletCache + bypass L1 (skipL1) + bypass consensus cache voting (v4.14.5) + bypass quota pre-check. Ne clear PAS GlobalPriceCache.

- **`packages/shared/src/cache-keys.ts`** — Registry canonique des clés cache (prix, scan, wallet) + helpers `cacheKey()` / `cacheKeyGsheet()` / `walletKey()`. Source unique pour toute construction de clé cache.

## Principes fondamentaux

1. **Ne JAMAIS écraser du cache valide** lors d'erreurs API
2. **Mode dégradé** : retourner données cached avec `[DEGRADED]`
3. **Circuit breaker** : 2 min de blocage après erreur quota
4. **Consensus RPC** : majorité stricte requise (`votes * 2 > total`) — 2/4 est un match nul, pas un consensus (v4.15.1)
5. **Cache = 1 vote** quand pas d'activité récente détectée (désactivé par forceFull ou activityForced v4.15.1)
6. **Préserver l'existant** : en cas de doute, garder les données cached plutôt qu'écraser avec des zéros

## Backup DB

Backup automatique quotidien à 03:00 via tâche planifiée Windows `WCORE_DB_Backup`.

| Fichier | Rôle |
|---------|------|
| `scripts/backup-db.js` | Export SQL de toutes les tables (Prisma $queryRaw) |
| `scripts/backup-db.ps1` | Wrapper PowerShell (set BACKUP_DATABASE_URL → node backup-db.js) |
| `scripts/setup-backup-task.ps1` | Crée la tâche planifiée Windows (backup quotidien 03:00) |
| `scripts/check-backup-freshness.ps1` | Vérifie que le dernier backup < 48h (anti-panne silencieuse) |
| `scripts/setup-backup-check-task.ps1` | Crée la tâche planifiée `WCORE_DB_Backup_Check` (quotidien 11:00) |
| `backups/` | Stockage des backups (rotation 7 jours) + `LAST_ERROR.txt` si check échoue |

**Backup manuel** : `powershell -File scripts/backup-db.ps1`

**Alerting freshness (P2-16, 2026-06-13)** : `scripts/check-backup-freshness.ps1` détecte les pannes silencieuses (incident 24/05→12/06 : 19j sans backup, `.env.backup` supprimé, `LastTaskResult: 2` ignoré). Si le dernier backup > 48h ou absent → écrit `backups/LAST_ERROR.txt` (cause + fix) + toast Windows + exit 1 ; self-clear quand sain. Tâche `WCORE_DB_Backup_Check` (quotidien 11:00). **Gotcha** : enregistrer l'action de tâche avec `-Command "& '<script>'; exit $LASTEXITCODE"` et NON `-File` — `-File` peut retourner un faux `3221225786` (0xC000013A) au teardown du process host. La source de vérité de l'alerte reste `backups/LAST_ERROR.txt`, pas le `LastTaskResult`.

**Restore** : `psql $DATABASE_URL < backups/wcore-backup-*.sql` (après avoir truncaté les tables)

**Vérifier** : `dir backups\` — doit avoir ≤ 7 fichiers.

**Attention** : l'ordre des tables est alphabétique. En cas de FK constraints au restore, désactiver les triggers ou restaurer les tables parentes d'abord.

**Gotcha `.env.backup` manquant (2026-06-12)** : la tâche planifiée exécute `scripts/backup-db-scheduled.ps1` qui lit `DATABASE_URL` depuis `scripts/.env.backup` (gitignoré via `.env*`). Si ce fichier est supprimé (ex: nettoyage de secrets), le script sort en `exit 2` → `LastTaskResult: 2` et **aucun backup pendant des semaines** (panne silencieuse, incident : aucun backup du 24/05 au 12/06). Diagnostic : `Get-ScheduledTaskInfo -TaskName "WCORE_DB_Backup"` (code 2 = `.env.backup` absent). Réparation : récupérer l'URL publique via `railway variables --service Postgres --json` → `DATABASE_PUBLIC_URL` (host `viaduct.proxy.rlwy.net`, PAS l'URL interne `postgres.railway.internal` inaccessible en local) et recréer `scripts/.env.backup` avec `DATABASE_URL=<url>`. Valider ensuite avec `Start-ScheduledTask` + vérifier `LastTaskResult: 0` (267009 = en cours).

## Gotchas

- **gid === 0** : `getSheetId()` retourne 0 pour le premier onglet → falsy en JS. Utiliser `gid != null` au lieu de `gid ?`
- **forceFull + quota** : forceFull bypass le L1 cache → plus d'appels HTTP que d'habitude. Ne pas lancer en fin de journée quand le quota est bas
- **Tokens LP non-stablecoin** : les tokens dont la seule liquidité est contre un quote non-USD (ex: TOKEN/CREATE) peuvent avoir des prix USD incorrects si le cache L1 est stale. Le skipL1 sur forceFull corrige ça (v4.13.9+)
- **GlobalPriceCache partagé** : toutes les chaînes partagent le même GlobalPriceCache. La dernière chaîne à sauver écrase le priceMap entier

- **_RpcLookup multi-RPC** : `_RpcLookup` dans 27_ACTIVITY_REFRESH.gs stocke RPCs/REST URLs par chaîne + VM type (v4.15.0). `getRpcs(chain)` retourne le tableau, `getVm(chain)` retourne "EVM"/"SVM"/"COSMOS". `BUILD_RPC_LOOKUP()` utilise `eval()` → ne fonctionne PAS en contexte trigger. - **ForceRefreshManager expiry** : le flag expire quand `last_full_scan_ms > requestedAt` (v4.15.2) — c'est-à-dire quand un full scan complet a eu lieu après la détection de TX. Safety TTL de 2h pour les flags orphelins. `BaseEngine.checkActivityForceRefresh` vérifie `state.lastFullScanMs` et clear le flag si le full scan est postérieur
- **SVM native balance sans consensus** : contrairement à l'EVM, le SVM Engine n'a PAS de consensus RPC pour le native balance. Si `getBalance` échoue mais les tokens réussissent, le nativeBalance=0 écrase le cache. Fallback ajouté en v4.14.10 : si RPC échoue et cache contient un balance>0, le cache est préservé
- **clasp run / Execution API** : OK depuis 2026-04-29. Token OAuth wcore-mcp (`652188583224-i1qqdjip5f598p7ft67rvib751en3d9l.apps.googleusercontent.com`) dans `.clasprc.json` avec TOUS les scopes (`script.scriptapp`, `script.external_request`, `script.storage`, `spreadsheets`, `userinfo.email`). Client secret dans `.playwright-mcp/`. Si le token expire, suivre la procédure dans "Autonomie sans MCP".
- **Trigger mensuel GAS** : pas de trigger "mensuel" natif dans Apps Script. Pattern : trigger hebdomadaire + staleness check interne (25j). `UPDATE_DYNAMIC_RPCS()` s'auto-skip si données < 25j → exécution effective ~28j
- **UrlFetchApp `deadline` et `timeout` NON-FONCTIONNELS** : ni `fetch` ni `fetchAll` ne respectent ces paramètres dans GAS. Aucune protection timeout native n'existe. Les erreurs quota ("Service invoked too many times") prennent ~10s par appel avant de throw. v4.15.4 contourne avec early-abort et budget temps dans batchWithConsensus (03_HTTP.gs + 05_RPC.gs)
- **ChainFactory config** : accéder via `chain.getConfig()`, PAS `chain._config` ni `chain.config` (undefined). Le global RPC est `RpcClient` (pas `Rpc`). `pickForConsensus` a `minRpcs=2` qui override le count demandé — truncate rpcList après si besoin de 1 RPC
- **UPDATE_DYNAMIC_RPCS consomme ~250 HTTP calls** : tester 241 RPCs + latency. Exécuter tôt après le quota reset (10h30 CET). Les appels de test/debug s'accumulent vite — le circuit breaker peut se tripper en milieu de session
- **GAS 6 min pour fonctions admin/trigger** : les fonctions non-@customfunction (UPDATE_DYNAMIC_RPCS, BUILD_RPC_LOOKUP, etc.) ont une limite de 6 min, pas 30s. `_testRpcLatency` (v4.15.5) utilise un budget de 4 min pour rester dans cette limite
- **`fetchAll` parallèle = latence trompeuse** : tous les RPCs partagent le même `t0` → tri par latence inefficace. v4.15.5 utilise des fetch individuels séquentiels pour une mesure réelle par RPC
- **return invisible dans l'éditeur GAS** : quand on exécute une fonction depuis l'éditeur Apps Script (bouton Run), les `return` ne sont PAS affichés dans les logs. Seuls `Logger.log()` et `console.log()` apparaissent. Pattern : ajouter `console.log()` dans les fonctions admin/diagnostic pour visibilité éditeur
- **safe-push brace warnings** : le compteur d'accolades de safe-push.ps1 est un simple compteur de `{`/`}` qui ne distingue pas strings/commentaires. Les warnings sur 04B_CACHE_WALLET.gs, 33_DYNAMIC_RPC.gs et 18_CLEANUP.gs sont des faux positifs (accolades dans des strings comme `"{"`, `"${"`)
- **GAS file load order** : les fichiers GAS sont chargés par ordre alphabétique (00_ avant 01_ avant 32_). Les IIFEs `(function(){...})()` s'exécutent immédiatement au chargement → les variables des fichiers ultérieurs ne sont pas encore définies. C'est pourquoi VERSION_SCANNER (00B) ne peut pas lire les `*_VERSION` variables et doit déférer à MODULE_AUTOREGISTER (32_)
- **WCORE_VERSION source unique** : `WCORE_VERSION` est défini UNIQUEMENT dans `01_INIT.gs` (objet avec MAJOR/MINOR/PATCH + toString()). Ne JAMAIS redéclarer dans 00_VERSION_REGISTRY.gs. L'enregistrement ModuleRegistry("WCORE") est fait dans 32_MODULE_AUTOREGISTER.gs (car 01_INIT n'est pas chargé quand 00_ s'exécute)
- **_mergeAssetsPreservingCached et activityForced (v4.15.33)** : le merge NE DOIT JAMAIS être skipé, même en forceFull. `activityForced` (WATCHDOG → `state.force=true` → `_forceFull=true` sur cacheObj) faisait skipper tout le merge → tokens absents du scan (consensus RPC fail) disparaissaient du cache. Fix : suppression du `if(force)return cacheObj` dans `_mergeAssetsPreservingCached` (04C_CACHE_GLOBAL.gs:982-984). Seuls les tokens **explicitement** rescannés avec balance=0 (présents dans newSet) sont supprimés. Repro : ZERO Network WBTC (3 RPC fragiles, 1 timeout, pas de majorité).
- **Balances microscopiques (E-12) sur single-RPC chains (v4.15.34)** : quand un RPC free tier limite les batchs (drpc.org = max 3 req/batch), le call `decimals()` peut ne pas être dans le batch réussi. Fallback à 18 sur un token à 6 décimales → balance divisée par 10^12 (2.83E-12 affiché comme 0). **Fix legacy** : ajouter `RPC.TOKEN_DECIMALS` dans la config chain pour les décimales connues. **Fix core** : utiliser `chain.RPC?.TOKEN_DECIMALS` comme override dans `getEvmWalletAssets` et `getErc20Metadata`. **Bump CACHE_VERSION** pour invalider les vieux caches.
- **Sentinel eth_getBalance (v4.15.34)** : certaines chaînes EVM retournent une valeur sentinel fixe pour `eth_getBalance` (ex: Tempo → 424242424242424... pour toutes les adresses). Cela pollue le cache native et fausse le total. **Fix** : `FLAGS.DISABLE_NATIVE_BALANCE = true` dans la config chain + `NATIVE_SYMBOL=""` pour skip la ligne native. Dans le core, garde-fou `disableNative ? 0n : await readNativeBalance(...)`.
- **Cache vote bloque correction decimals (v4.15.34)** : dans `_scanBatch` (09_SIMPLE_ROTATION.gs), le cache est ajouté comme "vote" au consensus RPC. Si le cache a été calculé avec un mauvais nombre de décimales (balance microscopique), il bloque la valeur correcte retournée par le RPC. **Fix** : `skipCacheVote` quand `cachedBalance > 0 && cachedBalance < 1e-12` ET pas de `decimals` valide dans metaMap. Ainsi le RPC seul peut corriger la balance sans être contrecarré par le cache erroné.
- **GM off-chain vs on-chain (v0.1.13+)** : le GM off-chain (POST /api/gm) met à jour le streak quotidien mais **n'empêche PAS** le GM on-chain. Le GM on-chain valide les deux (streak quotidien + streak per-chain). Le frontend doit refléter ça : le bouton "Off-chain" se grise après un GM off-chain OU on-chain (daily streak done), le bouton "On-chain" n'est jamais grisé par un GM off-chain — seulement par `sending` (tx en cours). L'événement `wcore-gm-done` sans `chain` = off-chain → GmButton l'utilise pour griser Off-chain uniquement. Avec `chain` = on-chain → ChaînCard grise cette chaîne spécifique.
- **ChainCard localStorage GM check** : utilise `wc_gm_onchain_date` (PAS `wc_gm_date`) pour vérifier si un on-chain GM a été fait aujourd'hui sur cette chaîne. `wc_gm_date` est commun à off-chain et on-chain → l'utiliser pour le check per-chain fausserait le résultat après un off-chain (le on-chain d'hier serait cru d'aujourd'hui).
- **GmContract.creatorAddress** : stocke l'adresse wallet du déployeur (lowercase). Ajouté en v0.1.13 pour le matching robuste dans my-contracts (ne dépend plus du CUID). Toujours renseigner `creatorAddress` à la création/upsert.
- **my-contracts matching** : utilise `OR: [{ creatorAddress }, { owner: { address } }]` pour trouver les contrats par adresse wallet, pas par CUID. Le background sync a un cooldown de 5 min par user.
- **POST /api/gm/contracts/deploy** : vérification on-chain obligatoire (txHash → receipt → factory → ContractDeployed event → status 0x1). L'adresse du contrat déployé est extraite depuis `topics[1]` de l'event `ContractDeployed`, PAS depuis `receipt.contractAddress` (qui est null pour les CREATE internes à la factory). Le frontend lit aussi l'adresse depuis le receipt log (pas `contractCount()` pour éviter les race conditions).
- **GM creator balance check (post-v0.1.19)** : `/api/gm/contracts/:id/balance` utilisait `contract.ownerId === req.user.id` (ID interne) pour déterminer si l'utilisateur est le créateur. Ce check échoue si l'utilisateur se reconnecte avec la même adresse wallet mais un ID session différent. Fix : utiliser `contract.creatorAddress?.toLowerCase() === userAddress` (adresse wallet), qui correspond à la réalité on-chain.
- **POST /api/gm/onchain** : transaction atomique via `prisma.$transaction`. Anti-replay (`[chainKey, txHash]` unique), score, streak, badges dans la même transaction. Le tip amount est décodé depuis l'event log (uint256 data) et persisté dans `OnchainGm.tipWei`. `OnchainGm.createdAt` doit être le timestamp de l'event `GmCheckedIn`, pas `now()`, sinon un retry/backfill tardif fabrique un GM "aujourd'hui". `rebuildChainStreakFromOnchain()` insère aussi `createdAt` depuis l'event et ne baisse pas un `UserChainGm.gmStreak` existant plus élevé.
- **Header On-chain vs ChainCard** : le Header sélectionne un contrat aléatoire multi-chaîne → se bloque après le 1er on-chain GM réussi (`alreadyOnChain`). Les GMs additionnels sur d'autres chaînes passent par les boutons **ChainCard** (bonus per-chain).
- **syncOnChainContracts overwrite protection** : avant d'upsert un contrat, vérifie si un `ownerId` existe déjà et appartient à un autre user → skip. N'écrase jamais un owner existant.
- **Off-chain double-click guard** : état `offchainSending` empêche les clics multiples pendant le POST off-chain.

## Gotchas WCORE Web - Security follow-up post-v0.1.19

- **CEX Binance/Bitpanda web (2026-06-13)** : le portage web des modules GAS `35_BITPANDA_SYNC.gs` / `36_BINANCE_SYNC.gs` garde les holdings CEX dans des tables separees (`cex_accounts`, `cex_holdings`) et les affiche via Profile > CEX. Ne jamais les injecter comme fausses chaines dans le scan on-chain. Contrairement a `wcore-gsheet`, le web NE fusionne PAS `USDC/TUSD -> USDT` ni `EUR/EURI -> EURC` : chaque symbole reste distinct et seul `trim().toUpperCase()` est applique. **Binance multi-user** : chaque utilisateur saisit SES PROPRES cles API read-only (`apiKey` + `apiSecret`) dans Profile. L'API WCORE les chiffre (AES-GCM) et les envoie au service `binance-relay` (Railway, IP non bloquee par Binance HTTP 451) qui SIGNE la requete HMAC avec le secret du user. Le secret ne touche jamais le navigateur. Bitpanda appelle l'API officielle server-side avec l'`apiKey` du user. Vars API requises : `RELAY_TOKEN` + `BINANCE_RELAY_URL` (ou `RAILWAY_SERVICE_BINANCE_RELAY_URL`) + `CEX_SECRET`.
- **binance-relay dual-endpoint (2026-06-13)** : le service `binance-relay` vit dans `wcore-gsheet/railway-relay/server.js` (source unique, deploye depuis la). DEUX endpoints : (1) `GET /binance?token=...` LEGACY pour `wcore-gsheet`/Apps Script — cles serveur fixes `BINANCE_API_KEY`/`BINANCE_API_SECRET`, symboles FUSIONNES (USDC/TUSD->USDT, EUR/EURI->EURC). NE PAS MODIFIER cet endpoint, `wcore-gsheet` en depend. Les deux partagent `RELAY_TOKEN`. Apres edit du relay, redeployer le service `binance-relay` depuis `wcore-gsheet`.
- **CEX dans `/home` + `/wallet` (2026-06-13)** : quand l'utilisateur est connecte, `useCexHoldings.ts` charge `/api/cex/accounts` et convertit chaque compte CEX en `ScanResult` synthetique (`address: cex:{provider}:{id}`, `isCex: true`, `chainKey: CEX_BINANCE`/`CEX_BITPANDA`). Les CEX apparaissent dans le selecteur `/wallet` et peuvent etre deselectionnes comme des wallets, mais ne declenchent JAMAIS de scan on-chain ni GM. Dans `/wallet`, le header externe garde `Bitpanda [CEX]` / `Binance [CEX]` sans icone redondante ; la carte interne affiche l'icone plateforme a la place de l'icone chaine et le badge `CEX` a la place du badge VM.
